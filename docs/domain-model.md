# OpenMarket Domain Model

## 사용자 그룹은 3개

플랫폼이 누구를 상대하느냐에 따라 도메인이 갈립니다.

- **구매자(Customer)** — 상품을 보고 사는 사람
- **셀러(Seller)** — 상품을 등록·판매·발송하는 사업자
- **관리자(Operator)** — 플랫폼 운영자, 셀러 심사·분쟁 중재·정책 관리

이 3개 그룹은 서로 다른 앱/어드민을 쓰고, 권한 체계가 다르고, 보는 데이터의 범위도 다릅니다. 백엔드에서는 인증·인가 모델을 처음부터 이 3개로 분리해서 설계해야 합니다.

## 도메인별 기능 요구사항

### 1. 회원 / 인증

- 구매자: 이메일·소셜 로그인(카카오/네이버/애플), 본인인증(휴대폰)
- 셀러: 사업자등록번호 검증, 정산 계좌 등록, 입점 심사 단계
- 관리자: 내부용, 2FA 권장
- 휴면 계정, 탈퇴(개인정보 보존 vs 즉시 삭제), 회원 등급/멤버십(WOW 같은 유료 멤버십)

### 2. 상품 카탈로그
   
오픈마켓의 핵심이자 가장 까다로운 영역.

- **카테고리**: 다단 트리(대→중→소→세). 카테고리별 필수 속성이 다름 — 가전은 제조일자, 식품은 유통기한
- **상품(Product) vs SKU**: 같은 티셔츠도 색상×사이즈 조합마다 SKU가 따로 있어야 재고/가격이 분리됩니다. Product 1개에 SKU N개 구조
- **이미지/상세**: 썸네일, 상세 페이지(HTML), 옵션별 이미지
- **상품 상태**: 작성 중 → 심사 대기 → 판매 중 → 품절 → 판매 중지
- **검색 메타데이터**: 검색어, 태그, 브랜드
- **금칙어/규제 상품 검수** (의약품, 주류, 의료기기 등)

### 3. 재고

SKU 단위 관리. 가장 동시성이 빡센 영역.

- 입고/출고/조정 이력
- 주문 시점 차감(예약) → 결제 완료 시 확정 → 결제 실패 시 복원
- 동일 SKU 동시 주문 경합 처리(락 vs 원자적 차감)
- 안전재고, 품절 알림

### 4. 검색 / 추천

- 키워드 검색 (RDB만으로는 한계 → Elasticsearch/OpenSearch)
- 자동완성, 연관 검색어, 오타 보정
- 필터(가격, 브랜드, 카테고리, 배송 옵션, 평점)
- 정렬(인기순, 낮은 가격순, 리뷰 많은 순, 신상)
- 개인화 추천(별도 시스템, 초기엔 인기 상품으로 대체)

### 5. 장바구니

- 비회원 장바구니(쿠키/디바이스 ID) → 로그인 시 병합
- 셀러별로 분리해서 표시 (배송비 계산이 셀러 단위)
- 가격/재고 변동 동기화 (담아둔 사이 가격 오를 수 있음)

### 6. 주문

복잡도가 폭발하는 곳입니다.

- **주문 분할**: 한 번 결제해도 셀러가 다르면 내부적으론 셀러별 주문 단위로 처리·배송됨
- **상태 머신**: 결제대기 → 결제완료 → 상품준비 → 배송중 → 배송완료 → 구매확정 / 취소 / 반품 / 교환
- **부분 취소**: 3개 상품 중 1개만 취소
- **반품/교환**: 사유, 귀책(소비자/셀러), 왕복 배송비 정산

### 7. 결제

- PG 연동(토스페이먼츠/나이스페이/KCP)
- 결제 수단: 카드, 계좌이체, 간편결제(카카오페이/네이버페이), 가상계좌, 적립금/쿠폰 혼합
- 결제 멱등성 필수 — 네트워크 끊김 시 중복 결제 방지
- 결제 취소 / 부분 환불
- 카드 정보 자체는 PG에 위탁이 안전 (PCI-DSS 회피)

### 8. 배송

- 배송지(주소록) 관리
- 셀러 직배송 / 자체 풀필먼트(로켓배송 같은) 구분
- 송장번호 입력 → 배송사 API로 추적 (스마트택배 같은 통합 API 활용)
- 배송 상태 변경 시 알림 트리거

### 9. 리뷰

- 구매 인증 리뷰 (산 사람만 작성 가능)
- 별점, 텍스트, 사진/동영상
- 부적절 리뷰 신고
- 셀러 답글
- 베스트 리뷰 / 추천순 정렬

### 10. 프로모션

- 쿠폰(%/정액, 최소 주문 금액, 사용 기한, 발급·사용 이력)
- 즉시 할인(상품/카테고리/시즌)
- 적립금(지급/사용/소멸)
- 기획전, 타임 세일

### 11. Q&A / 문의

- 상품 Q&A (셀러 답변)
- 1:1 문의 (관리자/셀러 라우팅)
- FAQ, 공지사항

### 12. 셀러 정산

플랫폼이 돈을 받아두고, 구매 확정 후 셀러에게 정산하는 모델.

- 거래 수수료 (카테고리별 차등)
- 정산 주기 (구매확정 + N영업일)
- 세금계산서, 매입/매출 신고 자료
- 환불·반품 시 정산 차감

### 13. 알림

- 채널: 앱 푸시(FCM/APNs), 이메일, SMS, 알림톡
- 트리거: 주문 단계 변경, 배송, 리뷰 답변, 마케팅
- 마케팅 알림 동의 별도 관리 (법적 요건)

### 14. 관리자 어드민

- 셀러 입점 심사
- 상품 검수 (위반 상품 회수)
- 분쟁 중재 (구매자 vs 셀러)
- 매출/거래 통계 대시보드
- 정책·약관 관리

### 비기능 요구사항

기능 못지않게 중요한 부분입니다.

- 데이터 정합성: 재고·결제·정산은 절대 깨지면 안 됨. 트랜잭션 경계 + 보상 트랜잭션(Saga) 설계 필요
- 트래픽 패턴: 평소 ≪ 피크(타임세일/광클). 캐시·큐·Read 분리 고려
- 가용성: 결제·주문은 SPOF 없게. 검색은 죽어도 상품 상세는 살아있어야 함
- 감사 로그: 누가 언제 무엇을 바꿨는지 추적 (특히 가격, 정산, 회원정보)
- 개인정보: 암호화 저장, 접근 로그, 파기 정책
- 보안: 결제·인증·관리자 어드민 망 분리, Rate Limiting

### 외부 연동

직접 만들지 말고 붙여야 하는 것들:

- PG: 토스페이먼츠 (문서 품질이 가장 좋음)
- 본인인증: NICE / KCB
- 사업자등록 진위 확인: 국세청 API
- 배송사 송장 추적: 스마트택배 / 굿스플로
- SMS/알림톡: 카카오 비즈메시지, NHN 토스트
- 푸시: FCM, APNs
- 이메일: AWS SES, Mailgun
- 검색엔진: Elasticsearch / OpenSearch
- 객체 스토리지: S3 (상품 이미지)

### MVP 우선순위 제안

- Phase 1 — **핵심 거래 플로우 (이게 안 되면 쇼핑몰이 아님)**
회원 / 상품 카탈로그 / 장바구니 / 주문 / 결제 / 배송 추적 / 셀러 입점 / 정산

- Phase 2 — 신뢰와 발견 검색 / 리뷰 / Q&A / 알림 / 쿠폰

- Phase 3 — 성장 추천 / 멤버십 / 기획전 / 마케팅 자동화 / 분쟁 중재 자동화
